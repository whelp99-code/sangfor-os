# UI/UX/AX Standards — Sangfor Agentic OS

## 0. 기준 문서와 적용 범위

이 문서는 내부 UX/AX 보조 기준이다. 최상위 제품 기준은
`docs/06_UX/Role_Based_UX_Dashboards.md`,
`docs/06_UX/Color_Agent_UX_Dashboard.md`, 그리고
`agentic_company_os_sangfor_final_package_v3_2` 패키지 문서다.

`3개 메뉴 only` 또는 역할 페이지 제거 방향은 현재 채택하지 않는다.
`sangfor-os`는 AIOS v1을 포함한 전체 프로젝트 통합 포털이므로,
Sales, Presales, Finance, Delivery, Support, Operator, Security의
role-based workflow를 유지한다. AI Workspace 컴포넌트는 업무 화면을
대체하지 않고 AI 활동, 근거, 승인 필요 상태를 보여주는 보조 패널로만 쓴다.

## 1. 핵심 원칙

### 1.1 One Job Per Page

한 페이지는 정확히 하나의 목적만 가진다. 사용자가 페이지를 열었을 때 "지금 여기서 뭘 해야 하지?"라는 질문이 나오면 디자인이 실패한 것이다.

현재 시스템의 가장 큰 문제는 한 페이지에 30개가 넘는 사이드바 항목과 수십 개의 위젯을 쑤셔 넣은 것이다. Sales 담당자가 내 파이프라인을 확인하려고 하면, 승인 현황과 Color Review와 견적 Diff가 한 화면에 뒤섞여 있다. 이는 집중력을 분산시키고 업무 속도를 떨어뜨린다.

올바른 예시:
- Sales 페이지 = "내 파이프라인 관리" ONLY
- Finance 페이지 = "승인 큐 처리" ONLY
- Support 페이지 = "Ticket 큐 응답" ONLY

각 페이지의 단일 목적은 페이지 제목 아래 1줄로 명시한다. 예를 들어, 페이지 제목이 "내 파이프라인"이라면, 그 아래에 "진행 중인 Deal을 관리하고 다음 액션을 결정하세요"라는 1줄 설명이 반드시 따라야 한다.

### 1.2 5초 규칙

페이지가 완전히 로드된 시점부터 5초 안에 사용자는 다음 세 가지를 알아야 한다.

1. 지금 이 페이지에서 무슨 일을 하는 곳인가
2. 현재 내가 봐야 할 가장 중요한 정보는 무엇인가
3. 다음으로 취해야 할 액션은 무엇인가

이 규칙을 지키지 않으면 사용자는 혼란스러워하다가 결국 엑셀이나 이메일로 업무를 처리하게 된다. 5초는 매우 긴 시간이 아니다. 페이지 로드가 2초 걸린다면, 실제로 사용자가 정보를 파악할 수 있는 시간은 3초에 불과하다.

측정 방법: 신규 입사자에게 페이지를 보여주고, 5초 후 화면을 가린 뒤 "이 페이지에서 무슨 일을 해야 하는지 말핸 봐"라고 질문한다. 답변 실패율이 10%를 넘으면 해당 페이지는 개선 대상이다.

### 1.3 클릭 3회 법칙

사용자가 원하는 정보에 도달하거나 원하는 액션을 실행하기까지의 클릭 수는 최대 3회를 넘지 않아야 한다.

현재 시스템에서 Sales 담당자가 특정 Deal의 견적 Diff를 확인하려면, 사이드바에서 "Sales"를 클릭하고, "Deal 목록"을 클릭하고, 특정 Deal을 클릭하고, "견적" 탭을 클릭하고, "Diff 보기"를 클릭해야 한다. 이는 4회 클릭이며, 이것이 반복되면 하루에 수백 번의 불필요한 클릭이 누적된다.

목표 클릭 수:
- 정보 조회: 최대 3클릭
- 액션 실행: 최대 2클릭 (예: 승인 버튼 → 확인 다이얼로그)
- 설정 변경: 최대 3클릭

클릭 수를 줄이는 구체적인 방법:
- 자주 찾는 정보는 대시보드 첫 화면에 배치
- 테이블 행 자체를 클릭하면 상세로 이동 ("자세히 보기" 버튼 불필요)
- 중요 액션은 행 위에 마우스를 올리면(hover) 바로 나타나는 툴팁 액션으로 제공

## 2. 낵비게이션 구조

### 2.1 사이드바 제한

사이드바의 항목 수는 절대 7개를 넘지 않는다. 인지 과학 연구에 따른, 사람의 단기 기억은 7개 플러스 마이너스 2개 항목을 처리할 수 있다. 30개 항목이 나에된 사이드바는 사용자가 원하는 메뉴를 찾는 데 수십 초를 낭비하게 만든다.

7개 항목이 넘는 경우, 하위 카테고리로 묶거나, 빈도가 낮은 메뉴는 "더 보기" 드롭다운으로 숨긴다. "더 보기" 안의 항목도 최대 5개로 제한한다. 그 이상은 별도 페이지로 분리하거나, 검색 기능으로 대체한다.

사이드바 항목 순서는 업무 빈도 기준으로 배치한다. 가장 자주 사용하는 메뉴가 상단에, 가장 드물게 사용하는 메뉴가 하단에 위치한다.

### 2.2 Role Switcher

화면 상단 중앙에 현재 역할을 표시하는 Role Switcher를 배치한다. 표시 형식은 "현재 역할: [역할명]"이다. 예를 들어, "현재 역할: Sales"라고 표시된다.

역할 변경은 클릭 1회로 가능해야 한다. Role Switcher를 클릭하면 드롭다운이 열리고, 다른 역할을 선택하면 즉시 전환된다. 전환 후에는 현재 페이지가 해당 역할의 메인 페이지로 자동 이동한다. 예를 들어, Sales 페이지에서 CEO로 전환하면 CEO의 Executive 개요 페이지로 이동한다.

지원 역할 목록:
- CEO
- Sales
- Presales
- Finance
- Delivery
- Support
- Operator
- Security

각 역할은 독립된 권한 집합을 가지며, 사용자는 여러 역할을 동시에 가질 수 있다. 하지만 한 번에 하나의 역할만 활성화할 수 있다.

### 2.3 메뉴 표준

모든 메뉴 항목은 아이콘과 한글 레이블을 함께 표시한다. 아이콘만 있거나 영문 레이블만 있는 경우는 허용하지 않는다.

아이콘은 Lucide Icons 라이브러리를 사용한다. 각 메뉴에 적합한 아이콘 예시:
- 내 Pipeline: GitBranch
- 승인: CheckCircle
- Color Review: Palette
- Ticket 큐: Ticket
- 시스템 상태: Activity
- 권한 변경: Shield

숫자 배지는 해당 메뉴에 처리가 필요한 항목 수를 즉시 표시한다. 예를 들어, "승인" 메뉴 옆에 빨간색 배지로 "5"가 표시되면, 처리 대기 중인 승인 요청이 5건 있다는 의미다. 배지가 0이면 배지를 아예 숨긴다. "0"이라고 표시하는 것은 시각적 노이즈다.

배지 색상 규칙:
- 1~3건: 파란색 배지 (정보)
- 4~9건: 주황색 배지 (주의)
- 10건 이상: 빨간색 배지 (경고)

### 2.4 Role별 메뉴 구성

각 역할별로 정확히 5~7개의 메뉴를 구성한다. 아래 테이블은 각 역할의 메뉴 우선순위와 알림 기준이다.

| 역할 | 메인 | 2순위 | 3순위 | 4순위 | 5순위 | 알림 배지 기준 |
|------|------|-------|-------|-------|-------|----------------|
| CEO | Executive 개요 | 승인 | Color Review | 리스크 관리 | | 위험 Deal 수 (마진 < 15%) |
| Sales | 내 Pipeline | 승인 | Color Review | 견적 Diff | | Follow-up 지연 건수 (7일 초과) |
| Presales | 내 Discovery | PoC | AI Draft 검토 | Sizing 현황 | | Sizing 미완료 건수 |
| Finance | 승인 큐 | 견적 Diff | 컨디션 리스크 | 마진 분석 | | 낮은 마진 건수 (마진 < 20%) |
| Delivery | 구축 현황 | SOW | 인수 일정 | 리소스 배정 | | 지연 건수 (예정일 초과) |
| Support | Ticket 큐 | SLA 현황 | 에스컬레이션 | 지식 베이스 | | SLA 임박 건수 (4시간 이내 만료) |
| Operator | 시스템 상태 | 워크플로우 | 장애 알림 | 로그 분석 | | Failed Jobs 수 (1시간 내 실패) |
| Security | 권한 변경 | 데이터 접근 | 감사 이상 | 정책 관리 | | 정책 위반 건수 (24시간 내) |

CEO의 "Executive 개요"는 회사 전체의 매출 파이프라인, 주요 리스크, 승인 대기 현황을 한눈에 보여주는 대시보드다. 승인 메뉴는 CEO가 최종 결재해야 하는 요청들을 모은다.

Sales의 "내 Pipeline"은 해당 Sales 담당자가 관리하는 모든 Deal의 현황을 표시한다. 기본 필터는 "내가 담당자인 Deal"이며, 다른 필터로 전환할 수 있다.

Presales의 "내 Discovery"는 현재 진행 중인 기술 검증 요청들을 관리한다. PoC 메뉴는 실제 Proof of Concept 진행 현황을 추적한다.

Finance의 "승인 큐"는 재무 관점에서 검토가 필요한 모든 승인 요청을 모은다. 견적 Diff는 견적 변경 이력과 차이점을 비교 보여준다.

Delivery의 "구축 현황"은 현장에서 진행 중인 설치 및 구축 프로젝트의 상태를 보여준다. SOW 메뉴는 Statement of Work 문서와 진행 상황을 연결한다.

Support의 "Ticket 큐"는 미처리된 지원 요청을 우선순위별로 정렬하여 표시한다. SLA 현황은 현재 진행 중인 Ticket의 SLA 위반 위험도를 색상으로 구분한다.

Operator의 "시스템 상태"는 Sangfor 장비 및 Agentic OS 인프라의 실시간 상태를 모니터링한다. 워크플로우 메뉴는 자동화된 작업의 실행 현황을 보여준다.

Security의 "권한 변경"은 사용자 권한 변경 요청과 이력을 관리한다. 데이터 접근 메뉴는 민감 데이터 접근 로그를 감시한다.

## 3. 페이지 레이아웃 표준 (4-Plate Pattern)

모든 페이지는 동일한 4단 구조를 따른다. 이 구조는 사용자가 어떤 페이지를 열어도 일관된 위치에서 정보를 찾을 수 있게 한다.

### 3.1 Plate 1 — 헤더

페이지 최상단에 위치한다. 왼쪽에는 현재 페이지 이름과 1줄 설명, 오른쪽에는 해당 페이지의 주요 액션 버튼 1~2개를 배치한다.

버튼은 2개를 초과하지 않는다. 3개 이상의 액션이 필요하면, 1~2차 액션으로 구분하거나 드롭다운 메뉴로 숨긴다.

버튼 우선순위:
- 1차 액션: 가장 중요한 액션. 파란색 배경에 흰색 글씨 (Primary Button)
- 2차 액션: 보조 액션. 흰색 배경에 회색 테두리 (Secondary Button)

예시 (Sales 페이지):
- 제목: 내 Pipeline
- 설명: 진행 중인 Deal을 관리하고 다음 액션을 결정하세요
- 1차 액션: [+ 새 Deal 등록]
- 2차 액션: [엑셀 내보내기]

### 3.2 Plate 2 — 상태 요약

헤더 아래 1줄로 현재 상태를 요약한다. 마지막 업데이트 시간을 포함한다.

형식: "현재 상태 요약 — 마지막 업데이트: [시간]"

예시:
- "총 12개 Deal 중 3개가 승인 대기 중입니다. 2개의 Follow-up이 7일 이상 지연되었습니다. — 마지막 업데이트: 14:32"
- "미처리 Ticket 8건. SLA 임박 2건. — 마지막 업데이트: 14:30"

이 1줄은 페이지의 전체 맥락을 제공한다. 사용자가 아래 내용을 자세히 보기 전에, 지금 상황이 얼마나 긴급한지 직관적으로 파악할 수 있어야 한다.

### 3.3 Plate 3 — 핵심 지표

상태 요약 아래에 최대 4개의 지표 카드를 가로로 배치한다. 4개를 초과하면 정보 과다가 된다.

각 카드는 다음으로 구성된다:
- 큰 숫자 (24px Bold)
- 레이블 (12px Regular, 회색)
- 변화량 (14px, 색상으로 구분)

변화량 표시 규칙:
- 증가: 빨간색 화살표 위 + 숫자 (부정적 지표일 경우) 또는 파란색 (긍정적 지표일 경우)
- 감소: 파란색 화살표 아래 + 숫자 (부정적 지표일 경우) 또는 빨간색 (긍정적 지표일 경우)
- 변화 없음: 회색 대시 기호 대신 "변화 없음" 텍스트

예시 (Sales 페이지):
- 1,240 | 총 Pipeline (억원) | +120 (지난주 대비)
- 12 | 진행 중 Deal | +2
- 3 | 승인 대기 | -1
- 2 | 지연 Follow-up | +1 (주의)

숫자는 천 단위마다 콤마를 찍는다. 예: 1,240. 소수점 이하는 필요한 경우에만 표시한다.

카드는 클릭 가능해야 한다. 클릭하면 해당 지표와 관련된 상세 뷰나 필터링된 테이블로 이동한다.

### 3.4 Plate 4 — 집중 영역

페이지의 주 목적이 드러나는 영역이다. 테이블, 리스트, 차트, 폼 등이 여기에 위치한다.

테이블이나 리스트는 한 화면에 최대 10개 행까지만 표시한다. 10개를 초과하면 "더 보기" 링크나 페이지네이션으로 분리한다. 사용자가 스크롤을 내리기 전에 페이지의 전체 구조를 파악할 수 있어야 한다.

테이블 컬럼은 최대 7개로 제한한다. 7개를 초과하면 가로 스크롤이 발생하거나 정보가 압축되어 읽기 어려워진다. 중요도가 낮은 컬럼은 "더 보기" 클릭 시 확장되도록 숨긴다.

각 행은 마우스를 올리면(hover) 배경색이 연한 회색으로 변경되어, 현재 내가 어떤 행을 보고 있는지 명확히 한다.

정렬은 기본적으로 가장 중요한 기준으로 설정한다. 예를 들어, 승인 큐는 "요청일시 내림차순", Pipeline은 "예상 마감일 오름차순"이다.

### 3.5 Empty State 규칙

데이터가 없는 경우, 절대 "No data"나 "데이터가 없습니다" 같은 메시지를 표시하지 않는다. 현재 시스템의 가장 큰 문제 중 하나가 Empty State의 부재다. 사용자는 데이터가 로딩 중인지, 진짜 없는지, 아니면 버그인지 혼란스러워한다.

Empty State는 반드시 다음 세 가지를 포함해야 한다.

1. 상황 설명: 왜 비어 있는지 설명
2. 다음 액션 제안: 지금 당장 무엇을 할 수 있는지
3. 액션 버튼: 클릭하면 해당 액션으로 연결

예시:
- "아직 등록된 고객이 없습니다. 첫 고객을 등록하면 Pipeline 관리를 시작할 수 있습니다." → [첫 고객 등록하기]
- "승인 대기 중인 요청이 없습니다. 모든 요청이 처리되었습니다." → [지난 승인 내역 보기]
- "진행 중인 PoC가 없습니다. 새 PoC를 시작해보세요." → [PoC 시작하기]
- "해당 조건에 맞는 Deal이 없습니다. 필터를 변경하거나 새 Deal을 등록해보세요." → [필터 초기화] [새 Deal 등록]

Empty State 화면은 중앙 정렬하며, 관련 일러스트나 아이콘을 함께 표시한다. 일러스트는 시스템 일러스트 라이브러리에서 선택하며, 각 도메인별로 3가지 이상의 일러스트를 준비한다.

로딩 상태도 Empty State와 비슷한 형태로 처리한다. "데이터를 불러오는 중입니다..."라는 메시지와 함께 스피너를 표시한다. 로딩이 3초 이상 걸리면, "로딩이 오래 걸리고 있습니다. 새로고침해보세요"라는 메시지와 [새로고침] 버튼을 추가한다.

## 4. 접근성 표준 (AX)

### 4.1 키보드 단축키

모든 주요 기능은 키보드로만 접근 가능해야 한다. 마우스 없이도 업무를 처리할 수 있어야 한다.

기본 단축키 표:

| 단축키 | 기능 | 적용 페이지 | 비고 |
|--------|------|-------------|------|
| Cmd + R | 역할 전환 | 전체 | Role Switcher 열기 |
| Cmd + K | 검색 | 전체 | 전역 검색 모달 열기 |
| 1 | 승인 | 승인 큐 | 현재 항목 승인 |
| 2 | 반려 | 승인 큐 | 현재 항목 반려 |
| Cmd + S | 저장 | 폼 페이지 | 현재 폼 저장 |
| Esc | 취소 | 전체 | 열린 모달/드롭다운 닫기, 폼 취소 |
| J | 다음 항목 | 테이블/리스트 | 다음 행으로 포커스 이동 |
| K | 이전 항목 | 테이블/리스트 | 이전 행으로 포커스 이동 |
| Enter | 선택/확인 | 전체 | 포커스된 항목 실행 |
| / | 검색 포커스 | 테이블 페이지 | 페이지 내 검색 박스로 포커스 |

단축키는 화면 하단에 툴팁으로 표시한다. 사용자가 "?" 키를 누르면 단축키 안내 모달이 열린다.

단축키 충돌을 방지하기 위해, 브라우저 기본 단축키(Cmd + T, Cmd + W 등)를 덮어쓰지 않는다. Cmd + R은 원래 새로고침이지만, 이 시스템에서는 역할 전환으로 재정의한다. 사용자가 3초 안에 추가 키를 누르지 않으면, 원래의 새로고침 기능이 실행되도록 할 수도 있다. 하지만 현재는 단순하게 Cmd + R을 역할 전환으로 고정한다.

### 4.2 WCAG AA 기준

모든 페이지는 WCAG 2.1 Level AA를 준수해야 한다.

명도 대비:
- 일반 텍스트: 배경과 4.5:1 이상의 명도 대비
- 큰 텍스트(18px Bold 이상): 배경과 3:1 이상의 명도 대비
- UI 구성 요소(버튼, 입력창 테두리): 배경과 3:1 이상의 명도 대비

현재 shadcn 기본 스킨의 일부 색상은 이 기준을 만족하지 못한다. 특회 회색 계열의 보조 텍스트 색상이 너무 연해서 가독성이 떨어진다. 모든 색상 조합은 접근성 검사 도구를 통해 확인해야 한다.

키보드 접근:
- 모든 대화형 요소(버튼, 링크, 입력창, 드롭다운)는 Tab 키로 포커스 가능해야 한다
- Tab 순서는 시각적 순서와 논리적 순서가 일치해야 한다
- 포커스가 화면 밖으로 나가지 않아야 한다

포커스 표시:
- 포커스된 요소는 2px 두께의 파란색(#005FCC) 아웃라인으로 표시
- 포커스 아웃라인은 요소에서 2px 떨어져 있어야 한다(offset)
- 포커스 표시는 절대로 display: none이나 outline: 0으로 제거하지 않는다

스크린리더:
- 모든 이미지에는 대체 텍스트(alt) 제공
- 아이콘 버튼에는 aria-label 제공
- 테이블에는 caption 요소나 aria-label로 테이블 목적 설명
- 상태 변화(성공, 오류, 로딩)는 aria-live 영역으로 알림

확대:
- 브라우저 200% 확대 시에도 모든 콘텐츠가 깨지지 않고 접근 가능해야 한다
- 가로 스크롤은 최소화한다. 1280px 너비에서 200% 확대 시 세로 스크롤만 허용

### 4.3 한국어 접근성

한국어 사용자를 위한 추가 접근성 기준이다.

폰트:
- 기본 폰트: Pretendard 또는 Noto Sans KR
- 폰트 크기: 최소 12px. 11px 이하의 폰트는 사용하지 않는다
- 줄 간격: 1.6. 한글은 영어보다 시각적으로 복잡하므로 넉넉한 줄 간격이 필요하다
- 자간: -0.01em. 한글은 기본 자간이 넓어 보이는 경향이 있어, 살짝 좁게 조정한다

숫자 표기:
- 천 단위마다 콤마 사용. 예: 1,234,567
- 날짜: YYYY.MM.DD 형식. 예: 2026.06.26
- 시간: 24시간제. 예: 14:30
- 화폐: 원 단위, 콤마 포함. 예: 12,340,000원

버튼 및 레이블:
- 모든 버튼은 한글로 표시한다. "Save", "Cancel", "Submit" 같은 영문은 절대 사용하지 않는다
- "저장하기"보다는 "저장"처럼 간결하게. 하지만 의미는 명확하게
- 상태 메시지도 한글로. "Loading..." 대신 "불러오는 중..."

색상 명:
- 색상만으로 정보를 전달하지 않는다. 색상과 함께 텍스트나 아이콘을 병행한다
- 예: 빨간색 배지 옆에 "긴급"이라는 텍스트를 함께 표시

## 5. 정보 계층

모든 텍스트는 5개 레벨 중 하나에 속한다. 페이지 내에서 일관된 계층 구조를 유지하면, 사용자는 자연스럽게 중요도를 인식한다.

| 레벨 | 용도 | 크기 | 두께 | 색상 | 예시 |
|------|------|------|------|------|------|
| Level 1 | 페이지 제목 | 20px | Bold | #111827 | "내 Pipeline" |
| Level 2 | 섹션 제목 | 16px | Semibold | #1F2937 | "진행 중인 Deal" |
| Level 3 | 카드 제목 | 14px | Semibold | #374151 | "이번 분기 요약" |
| Level 4 | 본문 | 14px | Regular | #4B5563 | "해당 Deal은 견적 승인을 기다리고 있습니다." |
| Level 5 | 보조 정보 | 12px | Regular | #9CA3AF | "마지막 업데이트: 14:32" |

페이지당 Level 1은 1개만 존재한다. Level 2는 최대 3개를 권장하며, 5개를 초과하면 페이지를 분리하는 것을 고려한다.

링크 텍스트는 Level 4와 동일한 크기이되, 파란색(#005FCC)과 밑줄로 구분한다. 방문한 링크는 보라색(#6B21A8)으로 변경한다.

에러 메시지는 Level 4 크기에 빨간색(#DC2626)으로 표시한다. 에러 메시지 아래에는 구체적인 해결 방법을 Level 5 크기로 추가한다.

## 6. 상태 표시 규칙

내부 시스템의 raw 상태명을 절대 사용자에게 노출하지 않는다. 현재 시스템은 "PENDING_APPROVAL", "QUOTE_DIFF_FOUND", "SLA_BREACH_IMMINENT" 같은 영문 상태명을 그대로 보여주고 있다. 이는 전문가조차 한눈에 이해하기 어렵다.

### 6.1 상태 매핑 테이블

내부 상태명과 사용자 표시명의 매핑 예시:

| 내부 상태명 | 사용자 표시명 | 색상 | 아이콘 | 설명 |
|-------------|---------------|------|--------|------|
| PENDING_APPROVAL | 승인 대기 | 주황색 | Clock | 최종 승인을 기다리는 상태 |
| APPROVED | 승인 완료 | 초록색 | CheckCircle | 승인이 완료되어 진행 가능 |
| REJECTED | 반려 | 빨간색 | XCircle | 승인이 거부되어 수정 필요 |
| QUOTE_DIFF_FOUND | 견적 차이 발생 | 주황색 | AlertTriangle | 견적 버전 간 차이가 감지됨 |
| SLA_BREACH_IMMINENT | SLA 임박 | 빨간색 | AlertOctagon | SLA 위반까지 4시간 이하 남음 |
| SLA_BREACHED | SLA 위반 | 빨간색 | AlertTriangle | SLA가 이미 위반됨 |
| FOLLOW_UP_OVERDUE | Follow-up 지연 | 빨간색 | Clock | 예정된 Follow-up 날짜가 지남 |
| POC_IN_PROGRESS | PoC 진행 중 | 파란색 | Beaker | Proof of Concept가 진행 중 |
| POC_COMPLETED | PoC 완료 | 초록색 | CheckCircle | PoC가 성공적으로 완료됨 |
| DEPLOYMENT_DELAYED | 구축 지연 | 주황색 | Truck | 예정된 구축 일정이 지연됨 |
| MARGIN_LOW | 마진 낮음 | 주황색 | TrendingDown | 예상 마진이 20% 미만 |
| MARGIN_CRITICAL | 마진 위험 | 빨간색 | AlertTriangle | 예상 마진이 15% 미만 |
| WAITING_SIZING | Sizing 대기 | 회색 | Ruler | 기술 검토 및 Sizing이 필요 |
| ESCALATED | 에스컬레이션 | 빨간색 | ArrowUpCircle | 상위 담당자에게 이관됨 |
| AUTO_REMEDIATION_FAILED | 자동 복구 실패 | 빨간색 | Wrench | 자동화된 복구 작업이 실패 |
| POLICY_VIOLATION | 정책 위반 | 빨간색 | ShieldAlert | 보안 정책 위반이 감지됨 |

### 6.2 색상 규칙

상태 색상은 전체 시스템에서 일관되게 사용한다. 같은 색상이 다른 의미로 사용되면 사용자는 혼란스러워한다.

| 색상 | 의미 | HEX 코드 | 사용 예시 |
|------|------|----------|-----------|
| 초록색 | 정상, 완료, 긍정 | #059669 | 승인 완료, PoC 완료, 정상 상태 |
| 파란색 | 진행 중, 정보 | #005FCC | PoC 진행 중, 진행 중인 Deal |
| 주황색 | 주의, 검토 필요 | #D97706 | 승인 대기, 구축 지연, 마진 낮음 |
| 빨간색 | 위험, 긴급, 위반 | #DC2626 | 반려, SLA 임박/위반, 정책 위반 |
| 회색 | 대기, 비활성, 정보 없음 | #6B7280 | Sizing 대기, 데이터 없음 |

색상은 배경색이 아닌 텍스트나 아이콘 색상으로 사용한다. 배경색으로 사용할 경우는 연한 톤으로 조정한다. 예를 들어, 빨간색 상태의 행 배경은 #FEF2F2(연한 빨강)을 사용하고, 텍스트는 #DC2626(진한 빨강)을 사용한다.

### 6.3 아이콘 규칙

아이콘은 상태를 색상과 함께 보완한다. 색맹 사용자도 아이콘 모양으로 상태를 구분할 수 있어야 한다.

| 아이콘 | 의미 | 사용 상태 |
|--------|------|-----------|
| CheckCircle | 완료, 정상 | 승인 완료, PoC 완료 |
| Clock | 대기, 시간 관련 | 승인 대기, Follow-up 지연 |
| XCircle | 거부, 실패 | 반려 |
| AlertTriangle | 경고, 주의 | 견적 차이, SLA 위반, 마진 위험 |
| AlertOctagon | 긴급 | SLA 임박 |
| Beaker | 실험/검증 | PoC 진행 중 |
| Truck | 배송/구축 | 구축 지연 |
| TrendingDown | 감소, 하락 | 마진 낮음 |
| Ruler | 측정/검토 | Sizing 대기 |
| ArrowUpCircle | 상승, 이관 | 에스컬레이션 |
| Wrench | 수리/복구 | 자동 복구 실패 |
| ShieldAlert | 보안 | 정책 위반 |

상태 배지는 아이콘 + 한글 텍스트를 항상 함께 표시한다. 아이콘만 있으면 의미가 모호해지고, 텍스트만 있으면 색맹 사용자가 구분하기 어렵다.

## 7. 검증 체크리스트

모든 페이지는 배포 전 아래 10가지 항목을 반드시 검증한다. 한 항목이라도 실패하면 해당 페이지는 배포할 수 없다.

### 체크리스트

1. **One Job Per Page**
   - [ ] 이 페이지의 단일 목적을 10단어 이내로 설명할 수 있는가
   - [ ] 페이지 제목 아래 1줄 설명이 있는가
   - [ ] 현재 페이지와 무관한 위젯이나 정보가 섞여 있지 않은가

2. **5초 규칙**
   - [ ] 페이지 로드 후 5초 안에 "무슨 일을 해야 하는지"를 알 수 있는가
   - [ ] 가장 중요한 정보가 화면 상단 50% 안에 있는가
   - [ ] 신규 사용자 테스트에서 90% 이상이 5초 안에 목적을 파악하는가

3. **클릭 3회 법칙**
   - [ ] 주요 정보에 도달하기까지 3클릭 이내인가
   - [ ] 주요 액션 실행까지 2클릭 이내인가
   - [ ] 사이드바 메뉴에서 목적지까지 2클릭 이내인가

4. **7개 메뉴 제한**
   - [ ] 사이드바 항목이 7개 이하인가
   - [ ] 각 메뉴에 아이콘과 한글 레이블이 모두 있는가
   - [ ] 메뉴 순서가 업무 빈도 기준으로 정렬되어 있는가

5. **한글 상태 표시**
   - [ ] 모든 상태가 한글로 표시되는가
   - [ ] 내부 raw 상태명이 노출되지 않는가
   - [ ] 상태별 색상과 아이콘이 일관되게 적용되는가

6. **액션 가능한 Empty State**
   - [ ] 데이터가 없는 경우 "No data"가 아닌 한글 메시지를 표시하는가
   - [ ] Empty State에 다음 액션 버튼이 포함되어 있는가
   - [ ] 로딩 상태와 Empty State가 명확히 구분되는가

7. **대비 4.5:1**
   - [ ] 모든 텍스트의 배경 대비가 4.5:1 이상인가
   - [ ] 큰 텍스트는 3:1 이상인가
   - [ ] 포커스 표시가 명확하게 보이는가

8. **키보드 전용 접근**
   - [ ] Tab 키로 모든 대화형 요소에 접근 가능한가
   - [ ] 단축키(Cmd+K, Esc 등)가 정상 작동하는가
   - [ ] 스크린리더로 주요 콘텐츠를 읽을 수 있는가

9. **첫 포커스 관리**
   - [ ] 페이지 진입 시 첫 포커스가 논리적인 위치에 있는가
   - [ ] 모달 열림 시 포커스가 모달 내부로 이동하는가
   - [ ] 모달 닫힘 시 포커스가 트리거 요소로 돌아가는가

10. **색상과 아이콘과 텍스트 병행**
    - [ ] 색상만으로 정보를 전달하지 않는가
    - [ ] 상태 표시에 아이콘이 함께 있는가
    - [ ] 상태 표시에 한글 텍스트가 함께 있는가

### 검증 프로세스

각 페이지는 개발 완료 후, 디자인 팀과 1명의 비개발 직군 사용자(예: 실제 Sales 담당자)가 함께 체크리스트를 검증한다. 10개 항목 모두 체크되어야만 운영 환경에 배포한다.

검증 기록은 페이지별로 문서화한다. 실패한 항목, 수정 내용, 재검증 날짜를 기록한다. 이 문서는 프로젝트의 docs/verification/ 디렉토리에 저장한다.

## 8. 현재 시스템 개선 로드맵

현재 시스템의 주요 문제와 개선 우선순위는 다음과 같다.

### 8.1 즉시 개선 (1주 이내)

- 사이드바 30개 항목을 7개 이하로 축소. Role별 메뉴 구성 테이블 기준으로 재구성
- 영문 상태명 전체를 한글로 교체. 매핑 테이블 적용
- Empty State 추가. 주요 10개 페이지부터 적용
- 숫자 표기에 천 단위 콤마 적용

### 8.2 단기 개선 (2~4주)

- shadcn 기본 스킨에서 명도 대비 4.5:1 미만인 색상 교체
- 키보드 단축키 5개 이상 구현 (Cmd+K, Esc, 1/2)
- Role Switcher UI 구현
- 4-Plate Pattern을 모든 페이지에 적용

### 8.3 중기 개선 (1~3개월)

- WCAG AA 전체 기준 충족 검증
- 스크린리더 최적화
- 한국어 접근성 기준 전면 적용 (Pretendard 폰트, 줄간격 1.6)
- 모든 페이지 5초 규칙 테스트 통과

이 표준 문서는 매월 1회 검토하여 개선한다. 사용자 피드백, 사용 데이터, 신규 기능 추가에 따라 내용을 업데이트한다.
